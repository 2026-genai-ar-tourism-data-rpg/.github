# 프롬프트 설계 v0 — 시나리오 콘텐츠 생성

> [시나리오_MVP_예시](./시나리오_MVP_예시.md)의 *텍스트 필드*(대사·선택지·지령·힌트·퀴즈)를 LLM이 채우게 하는 프롬프트.
> 핵심: **이야기는 LLM(동적), 구조·정답·카운터·종료는 코드(정적)** (8-D). 작성: kys / 2026-06-19

---

## 1. 3원칙 (모든 프롬프트 공통)
1. **근거 주입 + 환각 금지** — `[장소 정보]`(TourAPI overview)를 넣고 *"여기 근거해서만, 없으면 지어내지 마라"*.
2. **출력 JSON 강제** — 코드가 파싱·통제하므로 자유문장 ❌. JSON만.
3. **상태 주입** — `inventory`(연계 7-C)·`history`(멀티턴)·`stage`를 넣어 NPC가 인지.
- +길이·톤 제약(2~3문장, 도깨비 말투), +파싱 실패 시 **코드 폴백**.

## 2. LLM이 채우는 것 vs 코드가 통제하는 것
| LLM (동적·생성) | 코드 (정적·규칙) |
|---|---|
| 등장 대사 · 선택지 *문구* · 지령/힌트 *문구* · 퀴즈 *문구* | 분기 결과(플래그)·퀴즈 채점·카운터(N/M)·예산/쿠폰·연계·종료조건 |

## 3. 프롬프트 템플릿

### 3-A. NPC 대사 + 선택지 (분기 대화) — `/v1/dialogue/turn`
```
[시스템]
너는 위치기반 관광 RPG의 도깨비 NPC다.
- 정체: '{place_name}'의 {archetype} 도깨비. 모티프: {motif}. 말투: 어미 '~니라/~겠느냐', 감탄 '허허'. {persona}
- 반드시 [장소 정보]에 근거해서만 역사·사실을 말한다. 정보에 없으면 지어내지 않는다.
- 2~3문장. 군더더기·메타설명·이모지 금지.

[장소 정보]
{overview}

[플레이어 상태]
- 단계: {stage}   - 모은 것(연계): {inventory}   - 이전 대화: {history}

[과제] 플레이어에게 건넬 대사 1개 + 고를 선택지 2개. 선택지 중 하나는 항상 진행("의뢰를 받고 ~ 찾아 나선다"). 모은 단서가 있으면 자연스럽게 언급.

[출력] 아래 JSON만:
{"line":"<대사>","choices":[{"id":"a","text":"<선택지>"},{"id":"b","text":"<선택지>"}]}
```
> ⮑ 선택 후엔 같은 프롬프트에 `history`+선택을 더해 **재호출**(멀티턴) → 분기 대사 자동. (우리 `branching_service.py`가 이걸 함)

### 3-B. 지령 + 단계 힌트 (방탈출)
```
[시스템] 너는 '{place_name}'의 도깨비. 도깨비 말투. [장소 정보]에 근거.
[장소 정보] {overview}
[과제] 플레이어가 AR로 찾을 '{target}'(예: 기억석 조각) 지령 1줄 + 막힐 때 줄 힌트 2단계.
  힌트는 점점 구체적이되, 마지막 힌트도 위치를 통째로 다 말하진 않는다(방탈출 느낌).
[출력] {"order":"<지령>","hints":["<힌트1: 넓게>","<힌트2: 구체적>"]}
```

### 3-C. 퀴즈 (방탈출식)
```
[시스템] 너는 '{place_name}' 퀴즈 출제자. [장소 정보]의 사실만으로 출제. 추측·창작 금지.
[장소 정보] {overview}
[과제] 이 장소 역사/특징 기반 4지선다 1개. 난이도 {difficulty}. 정답은 장소정보에서 검증 가능해야 함.
[출력] {"q":"<문제>","options":["..","..","..",".."],"answer":<정답 index 0-3>,"wrong_hint":"<오답 힌트, 정답 직접노출 금지>"}
```
> ⚠️ **퀴즈는 LLM이 틀린 정답을 낼 수 있음** → MVP는 **핵심 퀴즈=수작업, 보조만 LLM**(8-C 하이브리드). 생성 퀴즈는 answer가 overview에 실제 있는지 코드/사람이 검수.

### 3-D. NPC 페르소나 합성 (8-B) — 노드당 1회
```
[시스템] 관광지 데이터를 도깨비 캐릭터로 변환. 규칙: 모티프 추출(① 대표인물 ② 상징물 ③ 기능 ④ 자연) → 아키타입(persona/guardian/trade/spirit).
[입력] title={title}, category={cat}, overview={overview}
[출력] {"name":"<○○ 도깨비>","archetype":"...","motif":"...","persona":"<말투·성격 2문장>","appearance_tags":["..."]}
```

## 4. 역사 인물 NPC 변형 (세종·이순신)
3-A에서 **정체/말투만 교체**:
```
- 정체: 광화문의 '세종대왕'. 위엄 있되 백성을 아끼는 자애로운 어조. 글·백성을 강조.
  반드시 [장소 정보] 속 사실에 근거. 현대어·메타 금지.
```
→ 도깨비 대신 실존 인물 톤. 나머지(근거·JSON·상태) 동일. 핵심 스토리 대사는 **수작업 권장**(사실성·연출).

---

## 5. 환각·품질 가드 (박준형)
- **근거 강제**: "[장소 정보] 밖 사실 언급 금지" + 출력 후 *overview에 없는 고유명사 나오면 리트라이*.
- **JSON 실패 폴백**: 파싱 실패 시 `line`=원문, 선택지=기본값(우리 `_parse` 이미 처리).
- **퀴즈 검수**: 생성 정답이 overview에서 검증 안 되면 폐기(수작업 퀴즈로 대체).
- **온도**: 대사 0.7 / 퀴즈·페르소나 0.3(사실 안정).
- **eval**: 장소 10곳 골든셋으로 "근거 이탈·말투 이탈·JSON 깨짐" 회귀 체크.

## 6. 우리 코드와 연결
- 3-A는 이미 `app/services/branching_service.py`로 구현됨(대사+선택지+JSON 파싱+inventory). → 여기에 3-B/3-C/3-D를 같은 패턴으로 추가하면 됨.
- 입력 `{overview}`는 `detailCommon2`/지역캐시에서, `{inventory}`는 앱이 전달.

---

## 7. 한 줄 요약
> **프롬프트 = [정체·말투] + [장소 정보 근거] + [플레이어 상태] + [과제] + [JSON 출력].**
> LLM은 *문구만*, 분기·정답·카운터·종료는 *코드가*. 핵심 대사·퀴즈는 수작업으로 안전판.
